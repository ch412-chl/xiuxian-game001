# CODEX_RULES

> 用途：作为本项目的 Codex 通用执行规范。  
> 目标：降低无效上下文读取与重复解释成本，在不明显影响使用体验的前提下稳定降低 token 消耗。  
> 适用范围：任意新会话、任意账号、任意任务类型。

## 1. 文件职责

### 1.1 `CODEX_RULES.md`
- 负责项目级长期规则
- 负责规定阅读顺序、真源识别、输出风格、维护流程
- 不记录某一轮会话的临时状态

### 1.2 `HANDOFF.md`
- 负责当前阶段接力
- 记录当前目标、当前状态、阻塞、风险、下一步
- 面向“下一个 AI 立刻接手”

### 1.3 各目录 `README.md`
- 负责该目录的局部说明
- 记录该目录用途、真源文件、阅读建议、当前有效结构
- 只保留当前有效结论，不写会话流水账

## 2. 新会话默认执行顺序

每次新会话默认按以下顺序执行：

1. 先读 `CODEX_RULES.md`
2. 再读 `HANDOFF.md`
3. 再读根目录 `README.md`
4. 再读 `product/产品需求/模块需求导航.md`
5. 再读本次任务直接相关目录下的 `README.md`
6. 再读任务所需真源文件
7. 若任务涉及编码，必须先读 `product/实现规范/实现前自检流程.md`
8. 最后才读具体代码文件

禁止默认行为：
- 不要一上来全仓扫描并通读所有文档
- 不要把历史归档、旧方案、报告文件默认当真源
- 不要在未确认范围前先做大规模重构

## 3. 真源识别规则

### 3.1 真源优先级
1. 当前明确标记的正式规格 / 正式配置 / 实现规范
2. `HANDOFF.md` 中已确认的当前有效结论
3. 目标目录 `README.md` 中的局部说明
4. 历史文档与分析报告，仅作参考

### 3.2 冲突处理
- 若总 PRD 与局部实现方案冲突：先以总 PRD 判断产品方向，再以当前实现规范判断代码落地方式
- 若 `README.md` 与真源文件冲突：以真源文件为准，并在本轮结束时更新对应 `README.md`
- 若 `HANDOFF.md` 与代码现状冲突：先核对真源，再修正文档，不可直接采信旧接力信息

### 3.3 默认非真源
以下内容默认不作为当前实现依据，除非任务明确要求：
- `历史归档/`
- 过期方案
- 历史报告
- 临时草稿
- 仅用于追溯的旧版本说明

## 4. 读取范围控制

### 4.1 最小读取原则
- 每次任务优先只读 1 到 3 个直接相关文件
- 先定位，再展开
- 先目录说明，再真源，再代码

### 4.2 推荐读取方式
1. 先看目标目录 `README.md`
2. 再看该任务对应真源
3. 再用搜索定位具体文件与字段
4. 最后只读命中的关键代码片段

### 4.3 避免高消耗行为
- 不为简单任务通读长文档全文
- 不为局部改动复盘整个项目
- 不在没有边界的情况下同时展开产品、代码、资源、自动化四条线

## 5. 任务输入规范

若用户未明确给出边界，默认从以下五项中自行补齐最小闭环：
- 任务目标
- 任务边界
- 当前真源
- 允许修改范围
- 验收标准

推荐任务格式：

```md
任务目标：
任务边界：
真源文件：
允许修改：
不要做：
验收标准：
```

## 6. 输出规范

### 6.1 默认输出风格
- 简单任务：直接给结果，少解释
- 中等任务：给结果、关键改动、验证方式
- 仅在必要时提供较长方案或分解

### 6.2 默认禁止
- 不重复复述用户原话
- 不输出大段低价值总结
- 不逐文件写流水账，除非用户明确要求

## 7. 当前项目阶段边界

当前项目默认按以下阶段边界执行：
- 当前主线优先级：产品方案、实现规范、数值真源、运行态验收
- 美术资源与贴图当前不是主线，不默认展开
- 涉及交互改动，先出方案，再改实现
- 若只是文档与真源补全，不默认扩展到代码与视觉层

## 8. 编码前强制门禁

凡是进入代码实现前，必须先确认以下 8 项是否齐全：
- 标题
- 文案
- 状态机
- 交互逻辑
- 数据真源
- 写回规则
- 依赖关系
- 微信运行态验收

若缺任一项：
- 先补文档或指出缺项
- 不得直接猜着编码

## 9. 执行边界

### 9.1 默认做法
- 一次只解决一个闭环问题
- 先定位范围，再实施修改
- 修改后做最小必要验证

### 9.2 默认不做
- 未经确认擅自扩任务
- 因顺手而改无关目录
- 把目录 `README.md` 写成会话纪要

## 10. 文档维护规则

### 10.1 每轮会话结束必须检查
- `HANDOFF.md` 是否需要更新当前状态
- 本次实际影响到的目标目录 `README.md` 是否需要增量维护

### 10.2 `HANDOFF.md` 更新条件
满足任一条件就更新：
- 当前主目标改变
- 当前阻塞或风险改变
- 真源判断改变
- 下一步优先级改变
- 本轮完成了会影响后续接手的关键事项

### 10.3 目录 `README.md` 更新条件
仅在以下情况更新对应目录 `README.md`：
- 本轮新增了该目录的正式真源文件
- 本轮修改了该目录的用途、结构或读取顺序
- 本轮确认了该目录新的有效约束
- 本轮清理了该目录中的过期口径

以下情况不更新目录 `README.md`：
- 只是小范围代码修 bug
- 只是临时讨论，没有形成稳定结论
- 只是 `HANDOFF.md` 的阶段性状态变化

### 10.4 `README.md` 更新要求
- 只写当前有效事实
- 不写“今天做了什么”
- 不写长过程
- 不写与本目录无关的规则

## 11. 根目录入口约定

项目根目录默认入口文件如下：

1. `CODEX_RULES.md`
2. `HANDOFF.md`
3. `README.md`

补充说明：
- `product/目录索引.md` 是文档地图，不是第一入口
- `product/` 下相关目录 `README.md` 属于第二层入口

若后续新增新的全局说明文件，必须在根目录 `README.md` 中补入口说明。

## 12. 落地目标

执行本规则后，应达到以下效果：
- 新会话不再从零开始理解项目
- 局部任务不再频繁读取无关上下文
- 目录知识沉淀回项目，而不是只留在聊天记录中
- `HANDOFF.md` 管接力，`README.md` 管结构，`CODEX_RULES.md` 管规范
