- `敌人配置总表.csv`
- `神通配置总表.csv`
- `成就配置总表.csv`

- `广告配置总表.csv`
- `道具配置总表.csv`

- 境界成长与关卡经验
- 敌人数值与门槛分析
- 神通伤害、CD 与解锁规则
- 成就目标与奖励

- 当前轻实现装备掉落

## 3. 仍待补充的需求 / 交互 / 数据支撑
### 3.1 待补充数据真源
1. 广告真实 adUnitId
- `广告配置总表.csv` 当前已建立结构，但开发 / 正式广告位 ID 仍是占位符。
- 这属于上线前阻塞项。

### 3.2 待补充交互验收
1. 历练页交互方向仍需用户最终拍板
- 当前实现规范已经给出“地下城堡式双页结构”方向。
- 但最终 UI 手感仍需在微信开发者工具中确认，不应只凭静态文档落地。

### 3.3 待补充文本层
1. 历练事件文案仍是通用模板
- `历练事件配置总表.csv` 当前提供的是正式结构与基础模板。
- 若后续追求世界观一致性，需要按 14 大境界补成“境界风味版事件文案”。

2. 图鉴简述目前已够支撑实现，但还不够支撑长线沉浸
- 当前更适合“先实现可读档案”，后续可再精修语气与文采。

3. 需要强制执行实现前自检流程
- 当前实现规范和主表已比较完整，但实现线程仍可能跳过检查直接编码。
- 需要把 `实现前自检流程.md` 作为编码前硬门禁。

## 4. 当前主链完整度判断
### 4.1 已成型
- 产品总 PRD
- 主链实现规范
- 数值真源
- 成就 / 图鉴 / 广告 / 神通 / 道具 / 破境数据真源
- 统一运行态验收总表
- 实现前自检流程
- 归档边界与接力入口

### 4.2 仍是风险点
- 代码运行时尚未完整消费所有正式真源
- 历练与探索的运行态体验尚未以微信开发者工具做完整点击回放
- `ExplorePanel` 的结构风险仍在

## 5. 建议优先级
1. 先让实现线程按现有真源把运行时读表链路接通
2. 再做微信开发者工具运行态 smoke 验证
3. 然后再补装备掉落池与更细的事件风味文本

## 6. 结论
当前产品链路已经从“规则零散”进入“主链有真源、支撑页有数据、自动化可读懂”的阶段。

现在最值得继续推进的，不再是新增规则，而是：
- 让代码真正消费这些真源
- 让微信开发者工具运行态验收形成闭环
