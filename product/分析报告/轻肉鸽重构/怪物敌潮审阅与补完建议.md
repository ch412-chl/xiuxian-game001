## 2. 已确认合理的部分

### 2.1 节奏骨架是对的
- `敌潮波次与首领结构方案.md` 已明确 5 段结构：开场波 -> 第一精英波 -> 第二常规波 -> 第二精英波 -> 首领波。

- 这说明“妖潮压迫 -> 精英门槛 -> 首领收束”的骨架已经存在，不是从零开始。

### 2.2 大境界主题方向是清楚的
- 结构方案已定义三种明确敌潮人格样板：
  - 野狼谷：扑压、围猎、突进

### 2.3 敌人主表的主线真源完整度高
- `敌人配置总表.csv` 已覆盖 42 个关卡，每关稳定 3 条模板：普通怪 / 小Boss / 大Boss。
- 每条模板具备名称、档案标题、血量、伤害、行动CD、技能、经验奖励等基础字段。
- 这张表足以支撑“主线按关卡读取敌人模板”，主问题不在主线敌人主表，而在轻肉鸽波次真源还没补齐桥接层。

## 4. 敌潮是否足够形成“妖潮压迫”

### 4.1 事实
- 当前波次表只有以下字段：`stageId`、`waveId`、`waveType`、`startSecond`、`endSecond`、`spawnPattern`、`enemyGroupId`、`spawnIntervalSec`、`maxAliveCount`、`eliteId`、`rewardNode`。
- 当前唯一完整波次只有 `LQ_EARLY_01` 这一关。
- 表中能看出节奏递进：
  - W1：`1.8s / 18只`
  - W2：`2.2s / 14只 / elite`
  - W3：`1.6s / 22只`
  - W4：`2.0s / 18只 / elite`
  - BOSS：`3.5s / 10只 / boss`

### 4.2 推演
- 这些字段只能证明“有时间切段”和“有场上数量控制”，不能证明一定能形成“妖潮压迫”。
- 原因很简单：压迫感不是由 `maxAliveCount` 单独决定，而是由“敌人从哪里来、怎么包、什么敌人混编、精英怎样改变走位空间、首领怎样把群怪压力收束到机制压力”共同决定。
- 当前表里最关键的压迫生成信息缺失，所以现状更像“排期脚本”，不是“敌潮体验真源”。

### 4.3 当前缺失项
- 缺 `enemyGroupId` 的正式定义表。
- 缺每波敌群组成规则：普通怪占比、远程占比、突进占比、召唤占比、护盾占比。
- 缺刷怪位置规则：边缘刷、夹击刷、背后刷、环形刷、前压刷各自的空间约束。
- 缺同屏上限按敌类拆分的限制，只给了总量上限，没有给“危险单位上限”。
- 缺精英入场后的压场规则：是否替代普通怪、是否带随从、是否触发一次性冲锋、是否改变刷怪口。
- 缺首领波中的“群怪是否继续刷、何时停止刷、何时转纯首领战”。

### 4.4 建议
- 把“压迫形成机制”单独真源化，而不是只保留 `spawnPattern` 文本代号。
- 每个 `spawnPattern` 至少要能回答三个问题：从哪里压、靠什么压、什么时候转压。
- 每个波次至少要能被微信运行态验收成一句可观察描述，例如“玩家在 10 秒内会被两侧狼群逼迫横移，而不是站桩清怪”。

## 5. 关卡主题与敌潮人格是否区分充分

### 5.1 事实
- 方案文档已定义关卡人格，但当前轻肉鸽波次表只落了一套狼系关卡 `LQ_EARLY_01`。
- 首领机制表只存在两条：`LQ_EARLY_01` 与 `ZJ_EARLY_01`。
- 波次表中 `spawnPattern` 使用的是 `edge_ring`、`front_pressure`、`edge_mix`、`cross_pressure`、`boss_entry` 这类通用英文代号，没有文档说明这些模式分别属于哪种主题人格。
- `敌人配置总表.csv` 内，同一大境界的 3 个普通怪名字不同，但技能结构几乎一致；同一大境界 3 个小Boss 技能结构也一致；同一大境界的大Boss 机制词条一致，主要变化是数值与档案标题。
- `妖物图鉴总表.csv` 只按“每个大境界 1 个普通怪 + 1 个Boss”收口，共 28 条，没有覆盖小Boss，也没有覆盖轻肉鸽精英。

### 5.2 推演
- 大境界之间的主题区分是有的，但“关卡主题 -> 敌潮人格 -> 实际波次脚本”这条链并未真源化，所以无法保证进入不同关卡时一定打出不同体感。
- 目前更像“世界观主题已定义，战斗语法未落表”。
- 尤其是图鉴把一个大境界的普通怪收口为单个代表，会进一步削弱“外围 / 中心 / 深处”这种阶段差异，也不利于玩家形成“这一关到底在打什么”的记忆点。

### 5.3 冲突点
- `野狼谷外围 / 中心 / 深处` 在敌人主表中是 3 个不同普通怪与 3 个不同小Boss，但图鉴只认 `灰鬃狼妖` 与 `噬月狼王`，没有承接 `裂齿狼妖`、`血鬃狼妖`、`铜爪狼将`、`啸风狼将`、`赤瞳狼将` 的身份。
- 轻肉鸽波次表使用通用刷怪模式名，没有写明“狼系为什么是围猎压迫，剑系为什么是扇面压缩，火系为什么是持续灼烧”，人格无法验收。
- 首领机制文案有主题方向，但没有把主题回写到波次脚本，例如“狼群召引”对应哪一段刷怪口变化、召唤物是否计入 `maxAliveCount`、是否与波次共存，全部缺失。

## 6. 当前主表与轻肉鸽波次真源的字段断裂

### 6.1 已确认的断裂

#### 6.1.1 `stageId` 没有回链主表键
- 波次表与首领表都用 `stageId`，例如 `LQ_EARLY_01`、`ZJ_EARLY_01`。
- `敌人配置总表.csv` 没有 `stageId` 字段，只有 `大境界`、`阶段`、`关卡`。
- 结果：轻肉鸽数据无法直接、唯一地映射到敌人主表中的中文关卡记录。

#### 6.1.2 `enemyGroupId` 没有真源
- 波次表引用了 `lq_wolf_pack_a`、`lq_wolf_pack_b`、`lq_mixed_pack_a`、`lq_mixed_pack_b`、`lq_boss_arena`。
- 当前指定文件中不存在任何一张“敌群组成表”来解释这些组到底包含什么敌人、数量比例、刷怪顺序、上限和是否混入特种怪。

#### 6.1.3 `eliteId` 没有主表
- 波次表引用了 `elite_lq_bonewolf_01`、`elite_lq_bladehound_01`。
- 当前不存在“轻肉鸽精英配置总表”或等价真源。
- 这会直接导致第一精英波、第二精英波无法被正式定义，更谈不上与图鉴、掉落、文案、资源联动。

#### 6.1.4 `bossId` 与敌人主表是弱关联
- 首领机制表有 `bossId` 与 `bossName`，但没有指向敌人主表具体哪一行的稳定键。
- 当前只能靠 `bossName` 人工猜测去对齐 `敌人配置总表.csv` 的大Boss 行，这不满足实现级真源要求。

#### 6.1.5 图鉴没有桥接字段
- `妖物图鉴总表.csv` 使用 `图鉴ID`，但敌人主表、波次表、首领机制表都没有引用这个字段。
- 当前图鉴系统与敌潮系统只是“名字相似”，不是“真源相连”。

### 6.2 结论
轻肉鸽当前不是“字段少一点”的问题，而是“缺桥”。没有桥字段，后续任何代码接入都会被迫写映射表、写临时判断、或者用名称硬匹配，最终又回到被规范明确禁止的“运行时发明数据”。

## 7. 图鉴 / 掉落 / 首领机制之间的依赖缺漏

### 7.1 图鉴侧缺漏
- 只覆盖 `普通怪` 与 `Boss`，未覆盖：
  - 小Boss
  - 轻肉鸽精英
  - 首领召唤物
- 没有字段说明“击败谁时解锁图鉴”：
  - 首次遭遇
  - 首次击败
  - 首次通关该 stageId
  - 首次进入首领波
- 没有字段说明图鉴应展示哪一套首领机制简介。

### 7.2 掉落侧缺漏
- 首领机制表里只有 `rewardDropGroup`，例如 `lq_boss_drop_01`、`zj_boss_drop_01`。
- 当前指定文件中不存在掉落包真源，也不存在掉落包字段定义文档。
- 没有写明 `rewardDropGroup` 与 `rewardNode` 的关系：
  - `upgrade_1/upgrade_2/clear` 是局内强化节点还是结算节点
  - `clear` 是否必然触发首领掉落
  - 掉落是否进入背包、图鉴、局外结算，完全没有写回规则

### 7.3 首领机制侧缺漏
- 只有 `mechanic1/2` 名称与一句描述，缺少可执行依赖：
  - 触发条件
  - 前摇/后摇
  - 警示方式
  - 作用范围
  - 持续时长
  - 冷却时间
  - 是否可打断
  - 是否与召唤、狂暴联动
  - 是否受当前波次继续刷怪影响
- 只有 `enrageThreshold`，但没有说明进入狂暴后会改哪些参数、切哪段状态、替换哪套招式。

### 7.4 写回规则缺漏
- 没有图鉴解锁写回规则。
- 没有首领击败后掉落写回规则。
- 没有精英波奖励节点写回规则。
- 没有“本局已见首领机制 / 已击败首领 / 已解锁图鉴”的状态归档口径。

## 8. 首领机制仍然空泛的部分

### 8.1 当前能确认的事实
- `噬月狼王` 已有两条机制名：`扑月`、`狼群召引`。
- `断岳剑主` 已有两条机制名：`残阵剑潮`、`御剑归鞘`。
- 结构方案要求首领至少有 2 个机制层，并要求有主题收束。

### 8.2 当前仍空泛的关键点
- 缺首领战状态机。
- 缺入场状态说明：登场演出是否锁玩家位移、是否清场、是否保留残余小怪。
- 缺招式阶段切换条件：血量阈值、时间阈值、召唤次数阈值。
- 缺召唤机制细项：召唤物来源、数量、生命周期、是否掉经验、是否计入图鉴。
- 缺狂暴后的具体变化：仅有 `0.3/0.35` 阈值，没有行为变化定义。
- 缺场地交互逻辑：安全区如何被压缩，边缘是否危险，是否要求玩家横移/绕背/打断。
- 缺招牌机制的失败反馈文案与验收标准。

- `eliteId` 没有名称、档案标题、机制文案。

- 首领缺阶段状态机。

### 10.3 缺交互逻辑
- 刷怪口如何形成夹击。

- 不同关卡主题是否一眼可辨。

## 11. 对主控的 3 条最关键建议

### 建议 1
先补“桥字段”而不是先加更多波次。优先统一 `stageId / stageName / 大境界 / 阶段 / 关卡` 的映射，不然后续所有轻肉鸽配置都会继续悬空。

### 建议 2
把 `enemyGroupId` 与 `eliteId` 从代号升级为正式真源。没有敌群组成表和精英表，就无法稳定制造妖潮压迫，也无法做图鉴、掉落、资源和验收闭环。

### 建议 3
把首领机制从“概念描述”补成“可验收状态机”。至少补触发条件、召唤规则、狂暴变化、图鉴/掉落写回和微信运行态验收，否则首领只会停留在文案层，不会成为真正的关卡收束器。
