# 轻肉鸽重构首轮并行审阅汇总结论

## 1. 文档定位
本文档用于汇总第一轮 3 个并行子代理的审阅结果，给主控、第二轮子代理与后续实现线程一个统一判断：
- 当前已经明确了什么
- 当前最危险的断裂点是什么
- 下一步应该先补什么，而不是各自扩方案

## 2. 审阅来源
- `战斗核心审阅与补完建议.md`
- `怪物敌潮审阅与补完建议.md`
- `成长强化审阅与补完建议.md`

## 3. 主控结论

### 3.1 方向已经正确
- 轻肉鸽版修仙自动施法战斗的大方向可以成立。
- “自动施法 + 波次生存 + 局内三选一强化 + 首领收束”的骨架已经足够成为当前版本唯一战斗重构方向。
- 当前不需要再争论“要不要做成轻肉鸽”，而应该进入“怎么补齐实现级缺漏”。

### 3.2 现在不能直接进入代码
- 三个子代理都得出了相同结论：**当前文档足以确定方向，但不足以安全进入实现**。
- 主要不是因为想法不清，而是因为：
  - 真源桥字段没补齐
  - 边界状态机没补齐
  - 局内强化层级还不够厚
  - 轻肉鸽运行态验收还没独立出来

### 3.3 当前最危险的不是数值，而是真源分裂
最危险的断裂点不是“某个数太大或太小”，而是同一件事在多个文件里有不同表达：
- 旧探索战斗 vs 新轻肉鸽战斗
- 总神通表 vs 轻肉鸽战斗神通表
- 敌人主表 vs 波次表 / 首领机制表
- 图鉴表 vs 精英 / 小Boss / 首领机制写回

如果不先统一真源映射，后续实现一定会重新发明数据关系。

## 4. 三个专项报告的共同断裂点

### 4.1 轻肉鸽战斗还缺边界状态机
当前主战斗流程已经有：
- `prepare`
- `wave`
- `elite`
- `upgrade`
- `boss`
- `settle`

但还缺关键边界状态：
- 撤离确认
- 广告复活决策
- 复活恢复
- Boss 过场
- 结算预览
- 结算正式写回

结论：**战斗总方案必须先补边界状态机，才能进入 UI 与代码。**

### 4.2 怪物真源缺“桥”
当前已经有：
- 关卡级敌人主表
- 轻肉鸽波次表
- 轻肉鸽首领机制表
- 图鉴表

但它们之间缺桥字段：
- `stageId`
- `enemyGroupId`
- `eliteId`
- `bossId`
- `atlasId`
- `dropGroupId`

结论：**先补桥字段与中间真源，再谈接入运行时。**

### 4.3 局内强化只有演示深度
当前强化池太薄，问题不是“还少几条词条”，而是结构上不够：
- 临时神通层为 0
- 学派专属被动层不足
- 流派内部成长链太短
- 通用强词条会过早压扁选择

结论：**成长方案必须先把“基础神通 / 临时神通 / 被动道法”三层拆清楚。**

### 4.4 旧探索链路与新轻肉鸽链路仍然并存冲突
当前冲突集中在：
- 进入关卡是先到旧探索页，还是直接进战场
- 旧“层数推进”与新“波次推进”谁是主进度
- 旧“单体怪物详情”与新“敌潮/精英/首领”如何共存

结论：**必须正式宣布：轻肉鸽主链是目标真源，旧探索战斗只作为旧实现约束，不再扩散。**

## 5. 第一优先级补项

### 5.1 必须先补的 4 类内容
1. **真源桥字段与中间主表**
   - 敌群组成表
   - 精英配置表
   - 首领机制桥字段
   - 图鉴 / 掉落桥字段

2. **战斗边界状态机**
   - 撤离确认
   - 广告复活
   - 复活恢复
   - Boss 过场
   - 结算预览 / 正式写回

3. **局内强化层级补完**
   - 临时神通解锁
   - 学派专属被动
   - 泛用词条权重
   - 强化池字段补足

4. **轻肉鸽微信运行态验收**
   - 旧 WT 不足以验新战斗
   - 必须新增轻肉鸽专属 WT-LR 用例组

### 5.2 在这些补项完成前，第二轮代理的限制
- 可以继续做方案
- 可以补文档
- 可以给出依赖和冲突说明
- **不能**直接推进到代码实现
- **不能**默认补旧探索页 UI
- **不能**假设现有数据表已经足够

## 6. 第二轮代理启动前必须满足的条件
第二轮的“交互与页面代理”“支撑系统与运营代理”启动前，至少要确认以下问题已经有唯一口径：

1. 进入关卡后是否直接进战场  
2. `runBag -> inventory` 的唯一写回时机  
3. `神通配置总表` 与 `轻肉鸽神通战斗配置总表` 的映射关系  
4. `stageId` 如何映射主线关卡  
5. 精英是否进入图鉴  
6. 首领掉落组是否已有正式真源  
7. 轻肉鸽战斗的微信运行态验收是否已经独立成组

## 7. 当前建议的主控动作

### 7.1 最优动作顺序
1. 主控先补“真源桥字段与中间主表”
2. 主控补“轻肉鸽战斗边界状态机”
3. 主控补“局内强化层级与字段结构”
4. 主控补“轻肉鸽专属微信运行态验收”
5. 再启动第二轮页面与支撑系统代理

### 7.2 当前不建议做的事
- 不建议先改 UI
- 不建议先做怪物表现层
- 不建议先拉长神通清单
- 不建议先调数值细节
- 不建议先改代码证明方案

## 8. 对后续实现线程的硬提示
- 当前轻肉鸽文档已经足以让实现线程理解方向
- 但还不足以让实现线程安全落地代码
- 若未先补齐：
  - 标题/文案
  - 状态机
  - 交互逻辑
  - 数据真源
  - 写回规则
  - 依赖关系
  - 微信运行态验收
  
  则实现线程**只能列缺失项和建议，不能直接猜着写**

## 9. 当前一句话结论
**第一轮并行审阅的结果不是“可以开工”，而是“方向已经定了，当前应该先补桥、补边界、补强化层，再开第二轮方案和代码”。**
