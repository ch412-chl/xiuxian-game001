# 战斗核心审阅与补完建议

## 1. 已确认合理的部分

### 1.1 方向层面已明确
- **事实**：`修仙轻肉鸽自动施法战斗总方案.md` 已明确新正式方向是“自动施法 + 波次生存 + 局内三选一强化 + 首领战”。
- **结论**：大方向清晰，且已经明确否定“旧 10 层事件探索页微修补”和“现代枪械换皮”，这为后续线程提供了明确边界。

### 1.2 单局骨架可作为统一模板
- **事实**：总方案已定义 `prepare -> wave -> elite -> upgrade -> wave -> upgrade -> boss -> settle` 的基础骨架。
- **结论**：这套骨架足以作为所有境界共用的“单局模板”，可以承载统一开发和统一验收。

### 1.3 局外与局内分层方向正确
- **事实**：总方案已区分“局外 3 神通装配”和“局内强化只影响本局，不写回永久成长”。
- **结论**：这一层级划分是对的，能够避免永久成长与局内临时强化混写。

### 1.4 runBag 分层意识是对的
- **事实**：总方案与旧实现规范都保留了 `runBag` / `inventory` 的分层思路。
- **结论**：临时战利品与永久纳戒分层是必须保留的核心规则，不应回退。

## 2. 仍缺失的信息

### 2.1 标题与文案缺失
- **事实**：总方案写了状态名，但没有写清新战斗页在 `prepare`、`upgrade`、`boss`、`settle_win`、`settle_fail` 下的标题文案、按钮文案、空态文案、失败提示文案。
- **影响**：实现线程会回退去复用旧探索页按钮与旧文案，测试线程也无法按文案验收。
- **建议**：补一节“轻肉鸽战斗页标题与文案规范”，至少覆盖：
  - 开场提示
  - 波次提示
  - 精英登场提示
  - 三选一强化标题与说明
  - 首领登场提示
  - 复活/认命/撤离提示
  - 通关/失败/撤离结算标题

### 2.2 战斗状态机不完整
- **事实**：总方案虽已有 7 个主状态，但缺少以下关键状态或边界规则：
  - `撤离确认` 未入状态机
  - `广告复活决策` 未入状态机
  - `广告复活成功后回到哪个战斗态` 未定义
  - `upgrade` 是否完全暂停战斗结算未定义
  - `boss` 前是否需要独立过场/清场未定义
  - “同帧击杀首领且我方死亡”的优先级未定义
  - 中断退出后重新进入的恢复规则未定义
- **影响**：实现时会出现状态跳转靠猜、复活后回错状态、Boss 前后奖励重复结算等问题。
- **建议**：把“主状态”和“边界状态”拆开定义，至少补齐：
  - `retreat_confirm`
  - `revive_offer`
  - `revive_resume`
  - `boss_intro`
  - `settle_preview`
  - `settle_commit`

### 2.3 交互逻辑缺失
- **事实**：总方案只写“关键时机使用储物袋道具”，但没有定义储物袋到底能在 `wave / elite / boss / upgrade / prepare` 哪些节点打开，以及打开后是否暂停战斗。
- **影响**：这会直接影响 UI、状态机和微信运行态验收。
- **建议**：补“储物袋开放节点与暂停规则”，明确：
  - 哪些状态允许打开
  - 打开后战斗是否冻结
  - 关闭后是继续原状态还是进入确认态

### 2.4 数据真源缺失到“字段层”
- **事实**：总方案已要求补 `局内强化配置主表 / 敌潮波次配置主表 / 首领机制配置主表 / 神通战斗配置主表`，但没有写清这些主表至少要支持哪些字段。
- **影响**：即使有表，也可能仍然不足以支撑 14 境界扩展与微信运行态验收。
- **建议**：至少补齐以下真源维度：
  - 波次所属境界 / 关卡 / 阶段
  - 敌潮脚本段落
  - 精英插入点
  - 强化池分组
  - 首领机制标签
  - 撤离开放节点
  - 奖励写回时机

### 2.5 写回规则口径不一致
- **事实**：
  - 总方案写的是：`settle_win` 后战利品先留在 `runBag`，返回流程再决定是否写入纳戒。
  - `探索与战斗实现级方案.md` / `历练主链实现级方案.md` 写的是：成功撤离或通关时直接写入 `inventory`。
- **影响**：这是典型的数据真源冲突；不先统一，代码和验收一定分裂。
- **建议**：主控必须指定唯一写回真源，明确“结算展示”和“正式写回”是不是同一步。

### 2.6 依赖关系缺失到“替换关系”
- **事实**：总方案写了依赖关系，但没有明确说明旧 `探索页`、旧 `遇敌待战`、旧 `怪物详情弹框`、旧 `层数推进` 是“继续并存”还是“被新战斗页替换”。
- **影响**：不同线程会出现“有人删旧链路、有人继续补旧链路”的并行冲突。
- **建议**：补一节“替换矩阵”，逐项标记：
  - 保留
  - 重命名
  - 下沉为子能力
  - 废弃

### 2.7 微信运行态验收缺失
- **事实**：总方案只列了 6 条轻肉鸽验收原则；当前总验收表仍主要围绕旧探索战斗链路。
- **影响**：新战斗即使做偏，也可能在旧 smoke 下“看起来通过”。
- **建议**：必须新增一组轻肉鸽专属 WT 用例，并标明旧 WT 哪些废弃、哪些保留、哪些要重写。

## 3. 关键冲突点

### 3.1 “进入探索页”与“进入战场”冲突
- **事实**：
  - 旧规范要求：历练 -> 探索页 -> 点击探索/自动推进 -> 遇敌待战 -> 战斗。
  - 总方案要求：进入关卡后直接进入战场，不再进入旧探索日志页。
- **结论**：这是当前最大结构冲突，不能两套同时为真。
- **建议**：主控必须明确“轻肉鸽关卡的第一页到底是不是旧探索页”；若不是，就应把旧探索页降级为旧方案文档，不再作为当前实现真源。

### 3.2 “层数推进”与“波次推进”冲突
- **事实**：
  - 旧规范核心变量是 `当前层数 / 总层数`。
  - 新方案核心变量是 `波次 / 精英 / 强化 / 首领`。
- **结论**：顶部信息结构、进度展示方式、验收口径都已冲突。
- **建议**：轻肉鸽关卡应改成“波次进度真源”，不要继续以层数作为主要战斗进度表达。

### 3.3 “单体敌人详情”与“敌潮战场”冲突
- **事实**：
  - 旧规范顶部显示的是“敌人名称 + 单体血条”，点开怪物详情弹框。
  - 新方案强调的是“敌潮推进 + 精英压制 + 首领主目标”。
- **结论**：旧的单体敌人信息模型不足以承载敌潮场景。
- **建议**：补充“敌方信息展示规则”，明确普通波次是否展示单体详情，还是只在精英 / 首领阶段展示目标详情。

### 3.4 “仅 3 神通格”与“局内临时神通/被动强化”冲突
- **事实**：
  - 旧规范底部固定 `1 行 3 列神通格`，只承接局外装配的 3 神通。
  - 新方案允许“解锁额外临时战斗神通”和“获得被动增益”。
- **结论**：当前展示结构无法说明临时神通/被动强化如何表现。
- **建议**：补充“局内强化外显规则”，至少明确：
  - 临时神通是否占 3 槽之一
  - 被动增益是否独立显示
  - 强化后 CD/触发方式如何对玩家可见

### 3.5 runBag 写回规则冲突
- **事实**：新旧文档对通关写回时机定义不一致。
- **结论**：这是数据层冲突，不只是文案冲突。
- **建议**：先统一写回真源，再允许实现线程编码。

## 4. 单局结构对 14 境界扩展的判断

### 4.1 当前判断
- **结论**：**单局骨架本身够用，但当前文档还不足以支撑 14 境界长期扩展。**

### 4.2 判断依据
- **事实**：当前单局结构只定义了统一流程，没有定义“境界差异到底靠什么被表达”。
- **推演**：如果后续只是把同一套 `wave -> elite -> upgrade -> boss` 套到 14 个境界上，而没有新增“境界主题层、强化池差异层、怪群机制层、首领机制层”，最终会退化成同一局换数值。

### 4.3 必须补的扩展承载位
- **建议**：至少补齐以下四类长期承载位：
  - `境界主题差异`：每个大境界的战场主题、怪群风格、首领主题
  - `关卡脚本差异`：同一骨架下每关的波次脚本模板差异
  - `强化池差异`：不同境界、不同流派的局内强化池分层
  - `节奏目标差异`：早期境界与后期境界的时长、压迫密度、Boss 占比边界

## 5. 建议补充的状态机与规则

### 5.1 建议采用“主战斗状态 + 边界状态”两层定义

#### 主战斗状态
1. `prepare`
2. `wave`
3. `elite`
4. `upgrade`
5. `boss`
6. `settle_preview`

#### 边界状态
1. `retreat_confirm`
2. `revive_offer`
3. `revive_resume`
4. `boss_intro`
5. `settle_commit`
6. `run_abort_or_recover`

### 5.2 必须补的切换规则
- **建议**：
  - 明确 `upgrade` 是否完全暂停战斗与计时
  - 明确 `revive_offer -> revive_resume` 回到哪一个主状态
  - 明确 `retreat_confirm` 允许出现在哪些节点
  - 明确 `boss_intro` 是否清理场上残余普通怪
  - 明确“我方死亡 / Boss 死亡同帧”优先级
  - 明确 `settle_preview` 与 `settle_commit` 的写回边界

## 6. 微信运行态必须补的验收项

### 6.1 必须新增的 smoke case

#### WT-LR-01 进入关卡即战场
- **前置条件**：存在轻肉鸽关卡入口
- **步骤**：从历练进入关卡
- **预期**：直接进入 `prepare` / 战场页；不出现旧 `自动推进 / 探索` 链路

#### WT-LR-02 单局主状态全链路
- **步骤**：完整打一局
- **预期**：至少按一次 `prepare -> wave -> elite -> upgrade -> wave/boss -> settle` 闭环，不跳态、不丢态

#### WT-LR-03 三选一强化暂停规则
- **步骤**：进入强化选择节点
- **预期**：战斗完全暂停；未选择前不继续刷怪、不继续掉血、不继续跑 CD

#### WT-LR-04 复活返回原状态
- **步骤**：在 `wave / elite / boss` 任一态死亡后触发广告复活
- **预期**：复活后回到正确战斗态，不回旧待战态，不重复触发开场或奖励

#### WT-LR-05 撤离节点合法性
- **步骤**：分别在 `prepare / wave / elite / upgrade / boss` 尝试撤离
- **预期**：只在文档规定节点可撤离，其余状态不给错误入口

#### WT-LR-06 Boss 阶段切换
- **步骤**：推进至 Boss
- **预期**：能明确感知 Boss 开始；普通敌潮是否停止/压缩与文档一致；不会仍沿用旧单体遇敌逻辑

#### WT-LR-07 runBag 写回分流
- **步骤**：分别验证通关、撤离、死亡离开三条路径
- **预期**：三条路径的 `runBag -> inventory` 写回结果与唯一真源一致，且不重复写入

#### WT-LR-08 局内强化不永久污染
- **步骤**：本局获得临时神通或被动强化，结束后再开新局
- **预期**：局内强化只影响本局，不污染永久神通装配与纳戒

### 6.2 必须重写或降级的旧验收项
- **建议**：
  - 旧 `WT-05 ~ WT-08` 基于“探索推进 -> 遇敌待战 -> 战斗”的验收，不能直接作为轻肉鸽主验收
  - 旧 `WT-07`、`WT-08` 需改写成轻肉鸽战场态验收
  - `微信开发者工具运行态验收总表.md` 的依赖真源应新增 `修仙轻肉鸽自动施法战斗总方案.md`

## 7. 对主控的 3 条最关键建议

1. **先定唯一真源，再允许实现**
- 当前最危险的问题不是“少写了几个状态”，而是新旧两套战斗主链同时存在且互相冲突。主控应先指定“轻肉鸽战斗总方案”为战斗主链唯一真源，并明确旧探索战斗文档哪些章节立即失效。

2. **先统一写回规则，再放行联调**
- `runBag` 到 `inventory` 的写回时机在新旧文档里不一致，这是数据级冲突。若不先统一，后续代码、测试、验收会三方各写一套口径。

3. **先补微信运行态 smoke，再继续拆子任务**
- 这次重构不是 UI 微调，而是主链替换。没有轻肉鸽专属 smoke case，后续各线程即使“局部看起来都对”，总链路也仍可能错。
