# T03 微信运行态验收补完建议

## 1. 文档定位
本文档用于补完轻肉鸽主链的微信开发者工具运行态验收建议，供主控、第二轮页面代理、实现线程和测试线程统一使用。

本文档**只做补完建议，不直接修改正式验收总表**。

## 2. 结论先行

### 2.1 当前正式验收总表不足以覆盖轻肉鸽主链
当前 [微信开发者工具运行态验收总表](file:///Users/cuihua/Documents/git/minigame-1/product/实现规范/微信开发者工具运行态验收总表.md) 的 `WT-01 ~ WT-20` 主要围绕旧探索链路，仍以：
- 历练 -> 探索页 -> 推进 -> 遇敌待战 -> 战斗
作为默认前提。

而轻肉鸽主链已明确改为：
- 历练 -> 战场 `prepare`
- `wave -> elite -> upgrade -> wave/elite -> upgrade -> boss -> settle`

因此必须新增一组独立 `WT-LR-*`，不能继续靠旧 WT 勉强验新战斗。

### 2.2 旧 WT 不应整体删除，而应分三类处理
- **保留**：仍适用于局外系统或非战斗主链的用例
- **改写**：逻辑目标仍然存在，但页面状态与交互路径已经改变
- **废弃**：其前提已经不再成立，继续使用只会误验

### 2.3 WT-LR 需要覆盖 4 个旧表未覆盖的断点
1. 进入关卡即进入战场，而非旧探索日志页
2. 三选一强化出现时，战斗必须硬暂停
3. 广告复活、撤离确认、Boss 过场、结算预览/写回必须进入运行态验收
4. 局内强化、临时神通、战场掉落不能污染永久状态

## 3. 建议新增的 WT-LR 用例组

## 3.1 组 A：进入战场与主状态骨架

| 编号 | 用例名称 | 前置条件 | 微信开发者工具实际操作 | 预期结果 |
| --- | --- | --- | --- | --- |
| WT-LR-01 | 进入关卡即战场 | 已从历练页选择轻肉鸽关卡；当前账号已解锁该关卡 | 在微信开发者工具中启动项目，进入历练页，点击轻肉鸽关卡的“进入战场/开始历练”按钮 | 直接进入战场页 `prepare` 状态；不得出现旧“自动推进/探索”底部按钮；顶部显示关卡名、波次信息或开场提示 |
| WT-LR-02 | 单局主状态全链路 | 可完整打一局；角色初始可存活到至少第一次强化 | 在微信开发者工具内完成一局，观察状态切换 | 至少出现一次完整链路：`prepare -> wave -> elite -> upgrade -> wave/elite -> boss -> settle`；不得跳态、丢态或回到旧“遇敌待战” |
| WT-LR-03 | 开场准备态最小时长 | 关卡可进入；本地设备性能正常 | 连续进入同一关卡两次，录屏或观察开场 | `prepare` 至少可被感知，不得瞬间闪过；但总时长不应拖到影响节奏 |

## 5. 与微信开发者工具实际操作的对应关系

## 5.1 建议的最小实测动作集
为避免 WT-LR 只停留在文档层，建议把微信开发者工具中的实际操作统一为以下动作组：

1. **启动与登录确认**
- 打开微信开发者工具
- 确认项目可编译、可预览、基础包可打开

2. **主链手动路径**
- 人物页 -> 历练页 -> 轻肉鸽关卡 -> 战场页
- 这是所有 WT-LR 的统一起点

3. **录屏或连续截图**
- 对以下节点必须录屏或连续截图：
  - `prepare`
  - 第一次精英波
  - 第一次三选一
  - Boss 过场
  - 通关或失败结算

4. **战场内关键操作**
- 点强化选项
- 点储物袋/道具
- 点撤离
- 点广告复活
- 点认命离开

5. **局后回查**
- 返回人物页
- 返回洞府
- 打开纳戒
- 打开图鉴/成就
- 对照永久状态是否被错误污染

## 5.2 建议的开发者工具验证顺序
1. 先跑 `WT-LR-01 ~ WT-LR-03`
2. 再跑 `WT-LR-04 ~ WT-LR-07`
3. 再跑 `WT-LR-08 ~ WT-LR-15`
4. 最后跑 `WT-LR-16 ~ WT-LR-20`

理由：
- 先确认战场入口和主状态
- 再确认强化和暂停
- 再确认边界态
- 最后验写回和永久状态

## 6. 实现前缺失项
以下信息若未补齐，实现线程**只能列缺失项与建议，不能直接猜着写**。

### 6.1 当前明确缺失项
1. **轻肉鸽正式边界状态机未形成独立主文档**
- 目前只有总方案与审阅建议中提到 `retreat_confirm / revive_offer / revive_resume / boss_intro / settle_preview / settle_commit`
- 但正式、唯一、可执行的边界状态机文档尚未看到

2. **撤离开放节点未唯一写死**
- 已有“在允许节点可撤离”的原则
- 但没有看到明确到 `prepare / wave / elite / upgrade / boss` 五态逐项允许/禁止的正式真源表

3. **复活回跳目标未唯一写死**
- 已明确“复活后必须回到原主状态”
- 但没有看到对应正式状态字段或恢复规则表

4. **Boss 过场是否清场未唯一写死**
- 文档里只有“普通怪数量压缩或停止新增”
- 还不是可直接实现的唯一规则

5. **settle_preview 与 settle_commit 的关系未唯一写死**
- 当前总方案写的是“本次战利品先写入 runBag，待返回流程决定是否写入纳戒”
- 但旧探索链路文档曾写过“成功撤离或通关时直接写入 inventory”
- 若未统一，不得直接编码结算写回

6. **同帧击杀首领且我方死亡的胜负优先级未定义**
- 这是运行态关键边界
- 若不定义，测试和实现都会各猜一套

7. **轻肉鸽专属广告次数限制未形成独立字段真源**
- 当前只看到广告配置总表和旧广告规则
- 但未看到“轻肉鸽局内广告次数、每局上限、同类广告互斥关系”是否已单独对齐到新主链

### 6.2 补充建议
1. 先补一份《轻肉鸽战斗边界状态机与写回规则》正式文档
2. 再把 WT-LR 正式并入《微信开发者工具运行态验收总表》
3. 并新增一张“战斗边界态可操作矩阵表”，至少列清：
- 当前状态
- 是否可开储物袋
- 是否可撤离
- 是否可复活
- 是否暂停战斗
- 是否可写回 runBag
- 是否可正式写回 inventory

## 7. 一句话结论
轻肉鸽主链当前最缺的不是更多页面稿，而是**一组能真实约束实现与测试的 WT-LR 运行态验收项**；在 WT-LR 正式并入总验收表前，旧 WT 只能部分保留，不能继续拿来验证新战斗主链。
