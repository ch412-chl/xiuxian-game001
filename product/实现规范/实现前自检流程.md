# 实现前自检流程

## 1. 目的
本文档用于约束下一个实现线程在写代码前，先检查当前功能在实现层面是否已经具备足够信息。若信息不完整，优先补文档，不允许直接靠实现猜测。

## 2. 适用对象
- 代码实现线程
- 自动化实现线程
- 联调线程

## 3. 总原则
任何功能在开始写代码前，必须先确认它是否已经具备以下 8 类信息：
1. 页面标题/功能定位
2. 展示文案
3. 状态机
4. 交互逻辑
5. 数据真源
6. 写回规则
7. 依赖关系与冲突约束
8. 微信开发者工具运行态验收项

若任一项缺失，不得直接实现，应先补产品文档或实现规范。

## 4. 自检清单
### 4.1 页面与功能定义
- 这个功能的唯一职责是什么？
- 玩家在这里的主决策是什么？
- 是否明确写了“不允许承担哪些职责”？

### 4.2 标题与文案
- 页面标题是否明确？
- 按钮文案是否明确？
- 空态文案是否明确？
- 成功/失败/冷却/缺失提示文案是否明确？

### 4.3 状态机
- 是否列出了完整状态？
- 是否写明每个状态显示什么、隐藏什么？
- 状态之间怎么切换？
- 是否定义了错误态、边界态、退出态？

### 4.4 交互逻辑
- 点击哪里发生什么？
- 返回路径是否明确？
- 是否存在需要确认层的操作？
- 是否明确禁止某些交互出现在某状态？

### 4.5 数据真源
- 是否有正式主表或正式文档作为唯一真源？
- 字段是否足够支撑实现？
- 字段的展示层与计算层是否区分清楚？
- 当前功能是否在运行时必须读表，而不是临时公式生成？

### 4.6 写回规则
- 哪些数据写入临时层？
- 哪些数据写入永久层？
- 成功、失败、撤离、中断、广告成功/失败分别怎么写回？

### 4.7 依赖关系与冲突
- 这个功能依赖哪些上游条件？
- 它影响哪些下游页面或状态？
- 是否和旧逻辑冲突？
- 是否明确禁止复活旧方案？

### 4.8 运行态验收
- 是否在 `微信开发者工具运行态验收总表.md` 中有对应用例？
- 是否知道这次改动至少需要回归哪几条 WT 用例？
- 是否存在交互改动但没有运行态验证要求的情况？

## 5. 必须暂停编码的情况
以下任一项成立时，必须暂停编码，先补文档：
1. 没有唯一真源
2. 状态机未定义完整
3. 标题/文案未定义
4. 只有 PRD 一句概述，没有实现规范
5. 有数据表，但字段不足以支撑运行时
6. 有交互改动，但没有微信开发者工具验收项

## 6. 推荐执行顺序
1. 读 `HANDOFF.md`
2. 读总 PRD
3. 读对应实现规范
4. 读对应主表
5. 对照本清单做自检
6. 自检全绿后再开始写代码

## 7. 自检输出模板
实现线程在开始编码前，建议先产出一段简短自检结论：
- 功能名称：
- 对应实现规范：
- 对应数据真源：
- 状态机是否完整：是 / 否
- 标题文案是否完整：是 / 否
- 写回规则是否完整：是 / 否
- 微信运行态验收项：WT-xx / WT-xx
- 是否允许开始编码：是 / 否

## 8. 当前版本强制要求
当前项目中，下列功能未经自检不得开工：
- 历练主链
- 探索与战斗
- 储物袋 / 纳戒写回
- 破境 / 渡劫
- 成就 / 图鉴
- 广告入口与奖励
- 敌人与关卡脚本真源接入
