# 实现规范

## 目录职责
本目录只存放“实现级方案”与“真源接入规范”。

设计目的：
- 将总 PRD 与实现方案分层
- 避免实现线程继续在 `产品需求/` 中混读总需求与实现细节
- 让接手线程明确：这里的文档直接约束代码实现，而不是背景说明

## 当前目录职责
- `历练主链实现级方案.md`：历练选关、战前整备与探索入口
- `探索与战斗实现级方案.md`：探索、遇敌待战、战斗、结算与储物袋
- `敌人与关卡脚本真源接入规范.md`：关卡经验表与敌人配置表如何接入运行时
- `人物页与洞府中枢实现级方案.md`：人物页、洞府页、纳戒入口与破境入口
- `成就与妖物图鉴实现级方案.md`：成就页、图鉴页、奖励领取与解锁反馈
- `广告系统实现级方案.md`：广告触发、频次、兜底与验收
- `破境与渡劫实现级方案.md`：破境入口、突破确认、渡劫演出、失败冷却
- `道具与装备轻实现级方案.md`：道具使用、储物袋 / 纳戒、装备轻实现
- `历练事件实现级方案.md`：探索事件页面、事件收益与状态切换
- `修仙轻肉鸽自动施法战斗总方案.md`：新战斗方向的总目标、状态机与系统边界
- `局内强化与神通成长方案.md`：局内三选一强化、流派与神通战斗成长
- `敌潮波次与首领结构方案.md`：敌潮、精英波、首领与关卡主题结构
- `轻肉鸽重构子代理321计划.md`：轻肉鸽重构的 3-2-1 子代理拆分与执行顺序
- `轻肉鸽微信运行态验收补充.md`：定义 WT-LR 轻肉鸽专项运行态验收

- `轻肉鸽支撑系统映射方案.md`：定义广告、成就、图鉴、runBag、破境与道具的单局映射关系

- `轻肉鸽战斗边界状态机与写回规则.md`：定义 revive / retreat / boss_intro / settle_commit 等边界状态与写回时序
- `轻肉鸽真源桥接与中间主表规范.md`：定义 stageId、敌潮、精英、首领、图鉴、掉落的唯一桥接关系
- `功能依赖关系总表.md`：统一说明轻肉鸽与主线各模块之间的输入、输出、写回与禁止反向耦合
- `缺漏信息判定示例.md`：示例说明哪些情况可以编码，哪些情况只能报缺口
- `实现优先级与重构顺序.md`：当前代码改造的依赖顺序与禁止并行项
- `微信开发者工具运行态验收总表.md`：当前版本统一运行态 smoke case
- `实现前自检流程.md`：编码前必须完成的信息完整性门禁

## 读取时机
出现以下任务时优先读取本目录：
- 调整主链交互或核心页面结构
- 规划轻肉鸽版战斗重构与子代理拆分
- 将真源配置接入运行时代码
- 修正代码与 PRD、UI/UE 的落地偏差
- 规划实现顺序、重构顺序或验收项

## 推荐读取顺序
1. 先读 `../产品需求/当前产品需求.md`
2. 再读 `../产品需求/模块需求导航.md`
3. 再进入与当前任务直接相关的实现级方案
4. 然后对照对应主表与运行态验收总表

## 真源说明
本目录属于实现约束层。
- 若总 PRD 与实现规范发生冲突，以总 PRD 为产品方向真源，以实现规范为当前版本代码落地约束
- 实现线程不得跳过本目录直接按个人理解修改主链交互

## 使用约束
- 不跳过本目录直接按个人理解改主链
- 不把本目录当作历史背景材料目录
- 若发现规范失效，先核对总 PRD、模块导航与主表，再修正文档与代码

## 维护规则
满足以下情况时，增量更新本 README：
- 新增了正式实现规范文件
- 调整了本目录文件职责或阅读顺序
- 确认了新的长期有效实现约束
- 清理了过期规范口径

以下情况不要更新本 README：
- 只是一次小修复
- 只是会话过程记录
- 只是 `HANDOFF.md` 的阶段变化
