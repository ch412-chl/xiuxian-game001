# 数值校准代理任务说明

## 1. 任务定位
本子代理只负责第三轮中的**数值校准层**，目标是在不改产品结构、不改页面职责、不改状态机、不改写回口径的前提下，把轻肉鸽版本的：
- 单局时长
- 普通怪 / 精英 / 首领 TTK
- 强化出现节奏
- 掉落强度
- 失败惩罚与复活收益
- 首日成长节奏
- 14 大境界之间的战斗差异感

收成可执行的正式数值口径。

## 3. 本轮必须完成的事
1. 检查当前轻肉鸽数值真源是否已足够支撑正式校准
2. 明确：
   - 单局目标时长
   - 普通怪 / 精英 / 首领 TTK 区间
   - 20 次强化在 20 波中的等级分布与 build 成型拐点
   - 掉落与复活收益强度
   - 首日与长期成长的节奏差
3. 找出会导致后期崩坏的数值断裂点
4. 产出第三轮正式结论，明确是否已具备进入实现校准阶段的条件

## 4. 正式交付物要求
优先**收紧现有数值与规范文档**，不要盲目扩文件。只有在确实无法避免时，最多新增 1~2 份正式文档。

至少需要完成：
1. 数值校准主结论
2. 当前数值真源缺口或异常点
3. 正式 TTK / 时长 / 强化 / 掉落口径
4. 与微信运行态验收的对照关系

## 5. 严格禁止事项
1. 不改代码
2. 不反向改页面职责
3. 不改主状态机名称与切换关系
4. 不改 `runBag -> settle_commit -> inventory` 写回链
5. 不把广告、图鉴、成就当数值补丁入口

## 6. 当前最该盯的风险
1. 单局时长是否过长或过短
2. 强化前后爽感落差是否不足
3. 普通怪 / 精英 / 首领是否只是单纯堆血
4. 失败惩罚与复活收益是否会破坏局内节奏
5. 14 境界间是否只有数值增大，没有战斗差异感

## 7. 统一门禁
进入代码实现前必须再次确认：
- 标题
- 展示文案
- 状态机
- 交互逻辑
- 数据真源
- 写回规则
- 依赖关系
- 微信运行态验收

若缺任一项：
- 只能说明缺失项

- 不可直接补文档
