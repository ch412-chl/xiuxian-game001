# 支撑系统与运营代理任务说明

## 1. 任务定位
本子代理只负责第二轮中的**支撑系统与运营层**，目标是把广告、成就、图鉴、纳戒、储物袋、道具、破境、短周期补救与新轻肉鸽主链重新映射，确保支撑系统不再挂在旧页式探索链上。

## 2. 本轮必须完成的事
1. 检查支撑系统与轻肉鸽战场主链是否存在旧页式残留口径
2. 收紧广告、成就、图鉴、纳戒、储物袋、道具、破境的入口、展示、写回与运行态验收
3. 明确每个支撑系统与：
   - `runBag`
   - `inventory`
   - `人物经验`
   - `settle_preview`
   - `settle_commit`
   的关系
4. 若发现缺少标题、文案、状态机、交互逻辑、数据真源、写回规则、依赖关系、微信运行态验收中的任何一项，只能列缺失项与补充建议，不可直接猜测

## 4. 正式交付物要求
本轮优先**收紧现有文档**，不要盲目增文件。只有在确实无法避免时，最多新增 1~2 份正式文档。

至少需要完成：
1. 支撑系统与运营层的主控结论
2. 与轻肉鸽主链冲突的口径清单
3. 已收口后的正式文档同步
4. 微信运行态验收补充或修订

## 5. 禁止事项
1. 不改代码
2. 不修改数值真源字段含义
3. 不恢复旧页式探索、旧挂机、旧洞府生产链
4. 不把广告设计成主循环核心
5. 不跳过 `settle_commit` 直接写永久库存

## 6. 当前最该盯的冲突点
1. 成就奖励是否可能错误进入 `runBag`
2. 图鉴解锁与首领登场 / 首次遭遇的映射是否唯一
3. 广告入口是否仍然残留在失败预览页
4. 破境、突破丹、inventory、runBag 是否仍有旧口径混用
5. 储物袋与纳戒的写回链路是否与轻肉鸽结算链一致

## 7. 统一门禁
进入代码实现前必须再次确认：
- 标题
- 展示文案
- 状态机
- 交互逻辑
- 数据真源
- 写回规则
- 依赖关系
- 微信运行态验收

若缺任一项：
- 只能说明缺失项
- 只能给补充建议
- 不可直接补文档
- 不可直接猜着写代码
