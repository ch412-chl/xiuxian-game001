# 轻肉鸽重构子代理 3-2-1 计划

## 1. 文档定位
本文档定义“修仙轻肉鸽自动施法战斗”重构时，子代理如何拆分、何时并行、何时串行，避免多人同时写方案导致冲突。

## 2. 总体原则
- 不是按部门名义拆任务，而是按依赖关系拆任务。
- 先定战斗，再定敌潮与成长，再定交互与支撑系统，最后做数值校准。
- 不允许一开始就把 UI、广告、成就、图鉴全部并行展开。

## 3. 第一轮：3 个并行子代理
### 3.0 当前状态
- **已完成**：第一轮 3 个并行子代理的首轮方案审阅已经完成。
- **对应审阅结果**：
  - `../分析报告/轻肉鸽重构/首轮并行审阅汇总结论.md`
  - 详细专项审阅已归档到 `../历史归档/过程产物/轻肉鸽首轮并行审阅/`
- **主控汇总结论**：
  - `../分析报告/轻肉鸽重构/首轮并行审阅汇总结论.md`
- **当前禁止**：在上述断裂点未收口前，不进入代码实现，不启动第二轮页面与支撑系统实现。

职责：
- 定义单局结构
- 定义状态机
- 定义胜负条件
- 定义角色控制与自动施法规则

交付物：
- `修仙轻肉鸽自动施法战斗总方案.md`

### 3.2 怪物与敌潮代理
职责：
- 定义怪物分类
- 定义波次结构
- 定义精英波与首领结构
- 定义不同关卡敌潮人格

交付物：
- `敌潮波次与首领结构方案.md`

### 3.3 成长与局内强化代理
职责：
- 定义局内三选一强化
- 定义神通局内成长规则
- 定义流派方向

交付物：
- `局内强化与神通成长方案.md`

## 4. 第二轮：2 个串行子代理
### 4.1 交互与页面代理
前提：必须吃完第一轮成果后才能开始。

职责：
- 定义战斗页、强化页、结算页结构
- 重写历练入口与战前整备入口
- 明确微信开发者工具中的页面验收项

当前状态：
- **已完成**
- 正式产物：
  - `轻肉鸽交互与页面总方案.md`
  - `轻肉鸽页面职责与交互验收矩阵.md`
  - `轻肉鸽战斗边界状态机与写回规则.md`
  - `轻肉鸽微信运行态验收补充.md`
- 本轮最终收口的 3 个口径：
  1. 强化候选不足 3 项时的保底补齐顺序
  2. 战前整备少于 3 神通时的 `待解锁` 占位与开局规则
  3. 怪物详情页对普通怪 / 精英 / 首领的展示字段边界

### 4.2 支撑系统与运营代理
前提：必须在战斗、敌潮、成长三者收口后再开始。

职责：
- 定义广告在轻肉鸽版本中的入口
- 重定义成就、图鉴、纳戒、储物袋如何映射到新主链
- 定义失败补救与短周期运营钩子

当前状态：
- **已完成**
- 结论文档：
  - `../分析报告/轻肉鸽重构/支撑系统与运营代理执行结论.md`
- 当前唯一剩余阻塞：
  - 广告真实 `adUnitId` 仍是上线前占位项

## 5. 第三轮：1 个数值校准代理
### 5.1 职责
- 统一校准：
  - TTK
  - 单局时长
  - 强化频率
  - 精英强度
  - 首领强度
  - 奖励节奏

### 5.2 为什么放最后
因为只有前 5 个方案都定了，数值校准才不会反复推翻前面的结构。

## 6. 为什么不是全部并行
若一开始就把产品、数值、战斗、怪物、UI、广告同时展开，会出现：
- 战斗节奏与怪物压力不匹配
- 数值与技能成长冲突
- UI 先画完，战斗主循环又改
- 支撑系统按旧主链设计，后续返工

## 7. 每个子代理的统一门禁
进入任何子代理前，都必须确认是否具备：
- 标题
- 展示文案
- 状态机
- 交互逻辑
- 数据真源
- 写回规则
- 依赖关系
- 微信运行态验收

若缺任一项：
- 只能列缺失项
- 只能给补充建议
- 不能直接猜写
- 不能直接进入代码
