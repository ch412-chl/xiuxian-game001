# 局内等级与升级候选池实现级方案

## 1. 文档定位
本文档定义轻肉鸽战斗中最容易被实现线程做粗的两层：
1. 局内等级如何增长
2. 升级后 3 选 1 候选如何生成

这份文档的目标不是重写“每级都会弹强化”，而是把这件事拆到可实现颗粒度，避免后续继续只靠一张 `轻肉鸽局内强化配置总表.csv` 硬撑全部逻辑。

## 2. 正式结论
当前版本必须把局内成长拆成四层真源：
1. `轻肉鸽局内等级配置总表.csv`
2. `轻肉鸽局内强化配置总表.csv`
3. `轻肉鸽升级候选池配置总表.csv`
4. `轻肉鸽神通战斗配置总表.csv`

各自职责：
- 局内等级表：定义每级需要多少战斗经验、每级给几次选择
- 局内强化表：定义“可被选中的升级项”本体
- 升级候选池表：定义“什么阶段、什么条件下，这些升级项能进池”
- 神通战斗表：定义神通本体与可升级目标

## 3. 第一性原则
如果只有“升级项主表”，没有“候选池规则”，实现线程就会把升级逻辑写成：
- 随机抽 3 个
- 没抽到再随便补

这会直接导致三个问题：
1. build 成型不可控
2. 流派手感忽强忽弱
3. 后期 20 次强化会出现大量空选项或垃圾选项

所以：
**升级项本体** 和 **候选池生成规则** 必须拆开。

## 4. 局内等级规则
### 4.1 唯一正式口径
- 单局等级上限：`20`
- 每次升级默认奖励次数：`1`
- 每次升级固定进入 `upgrade` 状态
- 升级不可囤积到波次结束统一发放

### 4.2 正式阶段划分
| 等级段 | 阶段名 | 作用 |
| --- | --- | --- |
| 1 ~ 5 | 起势期 | 尽快建立第一主技能与第一个生存抓手 |
| 6 ~ 10 | 构筑期 | 主流派成型，开始有清怪效率差异 |
| 11 ~ 15 | 扩展期 | 补第二输出轴或强生存轴 |
| 16 ~ 20 | 终局期 | 为高压怪潮与首领收束 build |

### 4.3 等级表最少字段
- `level`
- `needBattleXp`
- `cumulativeBattleXp`
- `rewardChoiceCount`
- `stageTag`
- `poolTier`
- `notes`

其中：
- `poolTier` 是候选池分层关键字段，不可省略

## 5. 升级候选池规则
### 5.1 候选池不是升级项主表
`轻肉鸽升级候选池配置总表.csv` 不定义效果数值，它只定义：
- 哪些升级项在什么阶段能进池
- 需要什么前置
- 权重是多少
- 是否属于流派保底

### 5.2 候选池分层
当前正式拆为四层：
1. `starter`
2. `core`
3. `synergy`
4. `finisher`

解释：
- `starter`：开局起势用，优先给能快速成型的基础输出 / 生存项
- `core`：主流派核心项
- `synergy`：联动项、辅助项、扩散项
- `finisher`：终局高价值补强项

### 5.3 每次升级的候选生成顺序
1. 读取当前等级对应的 `poolTier`
2. 读取当前已持有神通 / Buff / 升级项状态
3. 从合法候选中筛出：
   - 当前主流派强化
   - 可转流派强化
   - 通用保底强化
4. 按权重抽出 3 项
5. 若不足 3 项，按正式保底链补齐

### 5.4 正式保底链
若当前等级下合法候选不足 3 项，必须按以下顺序补齐：
1. 同流派可重复基础强化
2. 通用被动强化
3. 低阶保底强化

禁止：
- 直接返回 2 项或 1 项
- 抽出前置不满足的项
- 抽出已满级且不可重复的项

## 6. 升级候选池表最少字段
- `poolEntryId`
- `upgradeId`
- `poolTier`
- `candidateType`
- `targetSchool`
- `targetSkillId`
- `unlockLevelMin`
- `unlockLevelMax`
- `needOwnedSkillCount`
- `needTargetOwned`
- `needTargetUpgradeLevelMin`
- `weight`
- `isStarterGuarantee`
- `isGenericFallback`
- `notes`

## 7. 与现有主表的依赖关系
### 7.1 依赖 `轻肉鸽局内强化配置总表.csv`
它负责说明：
- `upgradeId` 是什么
- 效果文本是什么
- 具体数值改什么

### 7.2 依赖 `轻肉鸽神通战斗配置总表.csv`
它负责说明：
- `targetSkillId` 对应哪种神通
- 该神通属于哪个门类
- 是否允许被候选池优先照顾

### 7.3 依赖 `轻肉鸽局内等级配置总表.csv`
它负责说明：
- 当前等级属于哪一阶段
- 应该开放到哪个 `poolTier`

## 8. 展示与文案要求
升级弹层必须显示：
- 当前等级
- 当前波次
- 当前已持有主神通
- 3 个强化卡

每张强化卡必须至少有：
- 标题
- 类型标签
- 一句话说明
- 若存在前置关联，必须体现在文案或被规则过滤掉

## 9. 微信运行态验收
必须额外验证：
1. `Lv1 ~ Lv5` 是否明显更容易抽到起势项
2. 同一流派 build 是否能稳定成型，而不是连续多次被随机打散
3. 中后期是否仍有合法候选，不会频繁抽到无效卡
4. 20 次强化里是否能看到 starter -> core -> synergy -> finisher 的阶段感

## 10. 禁止事项
- 禁止继续把“候选池规则”写死在代码 if/else 里
- 禁止把 `poolTier` 省略成“靠等级硬判断”
- 禁止只维护升级项主表，不维护候选池表
- 禁止把候选池权重散落到多个文件中
