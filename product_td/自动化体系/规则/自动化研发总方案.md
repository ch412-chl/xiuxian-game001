# 修仙版地下城堡 · 自动化研发总方案

版本：v0.3
日期：2026-03-18
适用对象：策划、程序、测试、自动化任务

## 1. 目标
本方案用于将当前项目推进为“可持续自动化研发”模式，使自动化任务能够在产品文档尚未完全齐备的前提下，持续完成以下工作：
- 识别并补齐产品文档缺口
- 将 PRD 转换为 UI/UE、数值与代码实现任务
- 自动完成低风险代码实现与文档对齐
- 自动执行测试、可玩性检查、数值回归与结果沉淀
- 在低风险前提下自动提交 git 并 push
- 当前阶段不推进新美术生产，只维护资源待办与阻塞说明

## 2. 总原则
### 2.1 单一真源
自动化执行时，文档与数据真源优先级固定如下：
1. `product/prd_doc/current_prd.md`
2. `product/界面交互/界面交互规范.md`
3. `product/数值策划/关卡修为经验与境界升级表.csv`
4. `product/数值策划/敌人配置总表.csv`
5. `assets/资源映射/*.json`

若低层文档或代码与高层真源冲突，自动化必须优先修正低层内容。

补充说明：
- `敌人配置总表.csv` 已并入 `product/数值策划/`
- 原因是当前敌人表本质上是运行时数值真源，不是单独的展示配置目录
- 自动化后续检查敌人、经验、门槛与运行时接入时，统一按“数值真源”处理

### 2.2 阶段门禁
自动化每轮运行必须按以下顺序推进，不允许跳步：
1. 产品定义检查
2. UI/UE 对齐检查
3. 数值口径检查
4. 低风险实现
5. 自动测试
6. 可玩性检查
7. 结果回写
8. git 提交与 push

若某一阶段存在高风险阻塞项，则本轮停止后续实现，改为输出阻塞清单。

### 2.3 风险分级
自动化必须区分三类改动：

低风险：
- 文案同步
- 显隐规则同步
- 线框与 UI/UE 文档对齐
- 文档版本号、最近更新时间、修订日志回写
- 低风险样式和布局微调

中风险：
- 页面结构重排
- 数值微调
- 新增低复杂度页面入口

高风险：
- 核心玩法改向
- 战斗公式大改
- 成长曲线重算
- 多系统联动逻辑重构
- 美术风格大幅切换

高风险问题不允许自动直接落地，必须输出阻塞清单。

## 3. 自动化完整链路
每轮“完整链路自动化”必须完成以下 7 个核心模块，并把美术链路降级为后置维护项。

### 3.1 产品补全文档器
职责：
- 检查 PRD 是否缺少页面职责、状态机、边界、验收标准、异常处理
- 将口语化、主观描述收敛为客观规则
- 在低风险前提下直接补充文档

输出：
- 文档补全项
- 阻塞项列表

### 3.2 UI/UE 对齐器
职责：
- 检查 `当前产品需求.md`、`界面交互规范.md`、`界面交互线框.html` 是否冲突
- 检查页面分区比例、显隐状态、入口返回规则是否一致
- 修正明显不合理的空间利用和交互冲突

输出：
- UI/UE 修正记录
- 未决交互问题

### 3.3 数值回归器
职责：
- 检查成长、关卡、敌人、神通、突破和可玩性节奏
- 识别断档、过难、过易与资源闭环断裂

输出：
- 数值异常报告
- 建议调整值

### 3.4 低风险实现器
职责：
- 读取已定稿需求
- 完成低风险代码实现
- 不越权做高风险方向改动

输出：
- 已落地实现项
- 未实现高风险项

### 3.5 自动测试器
职责：
- 执行语法检查、配置解析、文档一致性检查、关键规则检查
- 若缺少测试基础设施，输出最小测试建设清单

输出：
- 测试通过项
- 测试失败项
- 阻塞级问题

### 3.6 可玩性检查器
职责：
- 判断是否达到当前阶段的“可玩最小标准”
- 识别误触、拥挤、反馈缺失、卡关与入口混乱

输出：
- 可玩性测试报告
- 阻塞体验问题

### 3.7 回写器
职责：
- 自动更新版本号、最近更新时间、修订日志
- 更新报告文档
- 在低风险前提下提交 git 并 push

输出：
- 修订日志
- 自动化报告
- git commit 记录

## 4. 结果沉淀文件
自动化每轮至少应维护以下产物：
- `product/自动化体系/规则/自动化执行清单.md`
- `product/自动化体系/报告/进度记录.md`
- `product/自动化体系/报告/本轮自动化报告.md`
- `product/自动化体系/报告/阻塞清单.md`
- `product/自动化体系/报告/可玩性测试报告.md`
- `product/自动化体系/报告/数值异常报告.md`
- `product/自动化体系/报告/资源待办.md`
- 报告中不应把 `.codex/automations/*/memory.md` 作为项目变更产物对外展示。

## 5. 自动更新时间规则
当自动化修改以下文件时，必须同步更新时间：
- `product/prd_doc/current_prd.md`
- `product/prd_doc/changelog.md`
- `product/界面交互/界面交互规范.md`
- 相关数值和资源方案文档

更新时间规则：
- `版本`：仅在正式规则变更时递增
- `最近更新`：每次有效修改后同步为当前本地时间
- `修订日志`：必须记录本轮核心变化，不允许只改正文不留痕

## 6. 后置美术链路
当前阶段暂停新美术资源生成，但保留资源目录、命名规范、映射表和资源待办，作为后续恢复美术自动化时的基础设施。

当前要求：
1. 不将立绘、战斗背景、UI 贴图生成作为主链阻塞项
2. 只维护 `assets/资源映射/*.json` 与 `product/自动化体系/报告/资源待办.md`
3. 美术问题仅记录，不在当前阶段直接触发出图

恢复美术链路时再启用：
- `product/自动化体系/规则/美术资源自动化规范.md`
- `product/自动化体系/规则/美术接入流程.md`
- `assets/README.md`

## 7. 当前推荐运行方式
当前项目推荐采用：
- 1 个“完整链路自动化”作为主任务，每 4 小时运行一次
- 其他专项任务保留配置，但默认暂停，避免并发改动相同文件

## 8. 当前自动化目标
当前阶段自动化目标不是“直接发布”，而是：
1. 补齐产品文档
2. 收敛 UI/UE 逻辑
3. 收敛数值口径并清理过期方案
4. 稳定低风险实现能力
5. 建立统一测试与可玩性检查能力
6. 将美术链路降级为后置待办，不阻塞逻辑闭环

当以上 6 项稳定后，再恢复美术自动化与发版自动化阶段。
