# T02 边界状态机与写回补完建议

## 1. 结论先行
当前轻肉鸽战斗的主流程方向已经够明确，但**边界状态、写回时序、广告兜底、Boss 过场**仍存在缺口与旧链路冲突。实现线程在进入代码前，必须先拿到一套唯一边界口径，否则一定会在 `runBag -> inventory`、死亡复活、撤离和首领切换处各写各的。

本文档只做两件事：
1. 把当前已能确定的唯一边界口径收拢成建议版本。
2. 把无法从现有真源中确认的部分明确列为缺失项，禁止猜测实现。

---

## 2. 完整边界状态列表

### 2.1 主战斗状态
1. `prepare`
- 进入关卡后的开场准备态。
- 用于展示关卡标题、战场标题、开场提示。
- 不进行敌潮推进，不进行掉血，不进行掉落结算。

2. `wave`
- 普通敌潮推进态。
- 允许持续刷怪、自动施法、击杀结算、经验即时入账。

3. `elite`
- 精英波战斗态。
- 仍属于战斗推进态，但当前主威胁从普通敌潮切换为精英目标。

4. `upgrade`
- 局内三选一强化态。
- 必须完全暂停战斗推进、刷怪、掉血、CD 视觉推进与掉落拾取结算。

5. `boss_intro`
- 首领登场过场态。
- 用于从普通敌潮/精英态切换到首领态。
- 必须有明确的清场、压场或保留规则，不能直接和 `boss` 合并成同一瞬间。

6. `boss`
- 首领战斗态。
- 首领为唯一主目标。
- 普通敌是否停刷、减刷或保留，必须由真源字段决定，不能运行时猜。

7. `settle_preview_win`
- 胜利预结算态。
- 只做展示，不做正式纳戒写回。

8. `settle_preview_fail`
- 失败预结算态。
- 只做展示失败结果与丢失结果，不做正式纳戒写回。

9. `settle_commit`
- 正式提交态。
- 负责把应写回的内容正式从 `runBag` 提交到 `inventory`，或正式丢弃。

### 2.2 边界状态
10. `retreat_confirm`
- 主动撤离确认态。
- 只允许从指定主状态进入。

11. `revive_offer`
- 死亡后的广告复活决策态。
- 只允许在本局尚未用过广告复活时出现。

12. `revive_resume`
- 广告复活成功后的恢复态。
- 用于恢复我方气血、恢复战斗时间流，并回到正确主战斗状态。

13. `run_abort_or_recover`
- 运行态中断恢复判定态。
- 用于未来处理切后台/异常重进时的恢复判断。
- 当前文档未提供完整恢复口径，故只能保留为占位边界状态，不能直接实现。

## 3. 状态切换图文字版

### 3.1 主链
1. `prepare -> wave`
2. `wave -> elite`
3. `elite -> upgrade`
4. `upgrade -> wave`
5. `wave -> boss_intro`
6. `boss_intro -> boss`
7. `boss -> settle_preview_win`
8. `settle_preview_win -> settle_commit`
9. `settle_preview_fail -> settle_commit`

### 3.2 失败链
1. `wave / elite / boss -> revive_offer`
2. `revive_offer -> revive_resume`（广告复活成功）
3. `revive_offer -> settle_preview_fail`（认命离开或广告失败后主动放弃）
4. `revive_resume -> 原死亡前主状态`

### 3.3 撤离链
1. `wave / elite -> retreat_confirm`
2. `retreat_confirm -> settle_preview_win`（传送符撤离成功或广告撤离成功）
3. `retreat_confirm -> 原主状态`（取消撤离）
4. `retreat_confirm -> 保持当前状态`（广告失败，不得吞掉入口）

### 3.4 Boss 过场链
1. `wave / elite -> boss_intro`
2. `boss_intro` 内必须先完成“普通敌清场规则判定”
3. `boss_intro -> boss`
4. 禁止 `wave` 直接跳 `boss`，也禁止首领与上一波普通敌无口径混战

### 3.5 中断恢复链
1. `wave / elite / upgrade / boss` 异常中断后进入 `run_abort_or_recover`
2. 当前缺恢复规则，不允许实现线程直接决定“重进回哪一态”

## 4. runBag / inventory 写回时序

## 4.1 已确认事实
- 击杀敌人获得的修为经验应即时入账，不依赖结算页。
- 战利品与临时掉落先进入 `runBag`。
- 旧链路与新链路在“通关时是否立刻写回 inventory”上存在冲突。

## 4.2 建议采用的唯一口径
### 4.2.1 经验
- `xpReward` 在敌人死亡时即时写入永久成长进度。
- 经验不进入 `runBag`。
- 死亡、撤离、通关都不回滚已获得经验。

### 4.2.2 物资与掉落
- 灵石、材料、装备、可掉落消耗品先进入 `runBag`。
- `settle_preview_*` 只展示 `runBag` 当前内容，不做正式写回。
- 只有进入 `settle_commit` 后，才允许：
  - 成功撤离/通关：`runBag -> inventory`
  - 最终失败/认命离开：`runBag` 丢失并清空

### 4.2.3 三条正式路径
1. **通关**
- 敌人死亡 -> 掉落写入 `runBag`
- 进入 `settle_preview_win`
- 玩家确认离开结算页
- 进入 `settle_commit`
- `runBag` 全部正式写入 `inventory`
- `runBag` 清空

2. **成功撤离**
- 当前已获得掉落保留在 `runBag`
- 进入 `retreat_confirm`
- 传送符撤离成功或广告撤离成功
- 进入 `settle_preview_win`
- 玩家确认离开结算页
- 进入 `settle_commit`
- `runBag` 全部正式写入 `inventory`
- `runBag` 清空

3. **最终失败 / 认命离开**
- 已获得掉落仍暂存在 `runBag`
- 进入 `settle_preview_fail`
- 玩家确认认命离开
- 进入 `settle_commit`
- `runBag` 不写入 `inventory`
- `runBag` 清空

### 4.2.4 当前禁止事项
- 禁止在 `settle_preview_*` 直接写回 `inventory`
- 禁止经验跟随 `runBag` 一起在失败时回滚
- 禁止通关和撤离分别各自实现一套不同写回函数
- 禁止“广告撤离成功直接回历练页且绕过结算预览”

## 5. 广告复活 / 撤离 / Boss 过场唯一口径

## 5.1 广告复活
### 建议唯一口径
1. 触发状态：仅 `revive_offer`
2. 触发前提：
- 当前主状态为 `wave / elite / boss`
- 我方气血归零
- 本局尚未使用广告复活
- 广告位启用且次数未耗尽
3. 成功结果：
- 进入 `revive_resume`
- 恢复到“死亡前主状态”
- 不回到 `prepare`
- 不回到旧探索待战态
- 不重开本波次
4. 失败结果：
- 停留在 `revive_offer`
- 不自动判失败
- 只能由玩家进一步选择认命离开
5. 单局限制：最多 1 次

### 当前缺失项
- 复活后恢复多少气血：现有真源未写死
- 是否有短暂无敌：现有真源未写死
- Boss 态复活后首领技能计时是否重置：现有真源未写死

## 5.2 广告撤离
### 建议唯一口径
1. 触发状态：仅 `retreat_confirm`
2. 触发前提：
- 当前主状态允许撤离
- 无传送符或用户主动选择广告撤离
- 广告位启用且本局未使用广告撤离
3. 成功结果：
- 进入 `settle_preview_win`
- 保留当前 `runBag`
- 后续通过 `settle_commit` 正式写回 `inventory`
4. 失败结果：
- 停留在 `retreat_confirm`
- 不吞掉撤离入口
- 不自动返回战斗
5. 单局限制：最多 1 次

### 当前缺失项
- `elite` 态是否允许撤离：广告方案与轻肉鸽总方案未完全写死
- `boss` 态是否允许广告撤离：现有文档未给唯一口径，不能猜

## 5.3 Boss 过场
### 建议唯一口径
1. `boss_intro` 是必需边界状态，不得删除。
2. `boss_intro` 负责：
- 显示首领登场标题与短提示
- 处理普通敌潮清场/压缩/停刷规则
- 锁定主目标为首领
3. `boss_intro` 期间：
- 不刷新新的普通敌
- 不开放强化选择
- 不开放储物袋
- 不开放撤离入口（除非后续真源明文允许）
4. `boss_intro` 结束后进入 `boss`

### 当前缺失项
- 普通怪是“立即清空 / 保留少量 / 全部转场后仍存在”：未有正式真源
- Boss 登场时长与文案：未有正式真源
- Boss 过场是否暂停 CD：未有正式真源

## 6. 实现前缺失项
以下缺失项当前不能由实现线程自行猜测，必须先补文档或真源：

### 6.1 复活恢复值
- 广告复活后恢复比例 / 固定值未写死。
- 是否附带短暂无敌、清弹、击退等恢复保护未写死。

### 6.2 撤离开放边界
- `prepare / upgrade / boss_intro / boss` 四个状态下是否允许撤离未统一。
- 当前只能确认 `wave` 允许、`retreat_confirm` 必须存在。

### 6.3 Boss 清场规则
- 首领入场前，普通敌是否清空、保留或减量，现有真源缺失。
- 这一项会直接影响掉落、经验、战场压力和复活位置，不能猜。

### 6.4 settle_preview 的交互结构
- 胜利预结算与失败预结算的按钮文案、确认按钮名称、是否可查看明细，现有文档未写死。

### 6.5 储物袋在轻肉鸽状态机中的开放节点
- 旧探索战斗方案允许在部分状态打开储物袋。
- 轻肉鸽战斗总方案只写了“关键时机可用”，未正式列出开放节点。
- 当前不能直接把旧节点照搬为新真源。

### 6.6 中断恢复规则
- `run_abort_or_recover` 仅能列为占位边界状态。
- 当前没有切后台、重进、断线恢复的正式口径。

### 6.7 同帧胜负优先级
- “首领死亡”和“我方死亡”同帧发生时，通关优先还是失败优先，现有文档未写死。

---

## 7. 最优落地顺序
1. 先在轻肉鸽总方案中正式补齐 `retreat_confirm / revive_offer / revive_resume / boss_intro / settle_preview / settle_commit`。
2. 再补 `runBag / inventory` 的唯一写回时序，并明确 `settle_preview` 与 `settle_commit` 的职责边界。
3. 再补广告复活、广告撤离、Boss 过场的正式字段与口径。
4. 最后再让交互与页面子代理、支撑系统子代理继续扩页面和按钮方案。

---

## 8. 一句话结论
T02 的核心不是“多补几个状态名”，而是**把战斗边界状态、写回时序、广告补救和 Boss 过场收成唯一口径，并明确告诉实现线程哪些地方现在还缺真源、绝对不能猜着写**。
