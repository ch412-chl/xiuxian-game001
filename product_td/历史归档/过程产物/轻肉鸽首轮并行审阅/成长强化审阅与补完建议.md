# 成长强化审阅与补完建议

## 1. 审阅范围与判断前提

### 1.1 已审阅文件
- `product/实现规范/子代理任务说明/成长与局内强化子代理任务说明.md`
- `product/实现规范/局内强化与神通成长方案.md`
- `product/实现规范/修仙轻肉鸽自动施法战斗总方案.md`
- `product/数值策划/神通配置总表.csv`
- `product/数值策划/轻肉鸽神通战斗配置总表.csv`
- `product/数值策划/轻肉鸽局内强化配置总表.csv`
- `product/产品需求/当前产品需求.md`

### 1.2 判断前提
- 以总 PRD + 实现规范 + 数值主表交叉判断，不以历史代码或旧方案猜测。
- 只评审“局外整备 3 神通 + 局内三选一强化”的闭环是否成立。
- 若缺标题、文案、状态机、交互逻辑、数据真源、写回规则、依赖关系、微信运行态验收，只列缺失项与建议，不补实现。

## 2. 已确认合理的部分

### 2.1 成长边界的大方向是对的
- 文档已经明确把“境界、角色经验、永久神通解锁、永久道具积累”归到局外，把“神通升阶、新临时神通、被动道法”归到局内。
- 文档也明确局内成长不写回永久数值，这一点和总战斗方案一致，方向清晰。

### 2.2 战斗主链已经给出局内强化插入点
- 单局结构里已经给出两次强化节点，长局可扩展到三次，能支撑轻肉鸽的最小爽点曲线。
- `upgrade` 已被纳入战斗状态机，而不是事后结算附带项，这一点是合理的。

### 2.3 三选一候选池规则具备基础约束
- 已定义“优先给当前装配神通相关强化”。
- 已定义“不刷 3 个完全无用选项”。
- 已定义“无效词条不得出现”。
- 已定义“避免连续多次只出同一流派”。

### 2.4 已经意识到性能与数值上限风险
- 文档提出“局内同时生效主动攻击神通不超过 5 个”，这是控制数值失控、画面污染和微信运行态性能的正确方向。

## 3. 当前强化池明显不足的部分

### 3.1 现有强化池只够演示，不够支撑 3 神通构筑变化
当前 `轻肉鸽局内强化配置总表.csv` 只有 6 条：
- `skill_upgrade` 4 条
- `passive` 1 条
- `support` 1 条
- `临时神通解锁` 0 条

这意味着：
- 大多数神通只有 1 条专属成长词条，成长深度不足。
- 只要一局出现 2 次强化，玩家很快就会看到重复项或明显最优项。
- 想让“同样装 3 个神通，但这一局走不同构筑”成立，当前池子不够厚。

### 3.2 强化池没有覆盖文档要求的三层结构
文档要求局内强化至少包含：
- 已装配神通强化
- 新临时神通解锁
- 被动增益

但当前主表只有前两层中的一半：
- 有神通强化
- 有极少量被动/辅助
- 没有任何“新临时神通解锁”真源

这会直接导致：
- 方案里说“局内可拿到额外临时战斗神通”，表里却没有可落地数据。
- 构筑只能围绕 3 个初始神通做数值叠加，缺少中途转向和补位空间。

### 3.3 强化覆盖面不足，导致候选池容易塌缩
当前 6 个战斗神通里：
- `御剑术` 有专属强化
- `火弹术` 有专属强化
- `冰锥术` 有专属强化
- `天雷引` 有专属强化
- `清心诀` 有专属辅助强化
- `剑气斩` 没有专属强化

结果是：
- 若玩家局外装配含 `剑气斩`，其成长空间天然比其他神通浅。
- 若玩家未携带 `清心诀`、`天雷引`，候选池可用项会进一步变窄。
- “优先给当前装配神通相关强化”这条规则，在当前池深下很容易失效成“只能反复给泛用词条”。

### 3.4 当前池子不足以支撑 5 个流派目标
文档想支持：
- 御剑穿透流
- 火法爆裂流
- 冰法控场流
- 雷法点杀流
- 护体续航流

但当前表实际只有：
- 御剑相关神通 2 个
- 火法 1 个
- 冰法 1 个
- 雷法 1 个
- 心法 1 个

并且多数流派只有“1 个基础神通 + 1 条强化”。这不叫流派，只能算“神通标签”。

## 4. 流派冲突与同质化风险

### 4.1 目前更像数值叠层，不像构筑分化
当前强化几乎都是：
- 数量增加
- 半径增加
- 减速增强
- 连锁增加
- 全局冷却缩短
- 回复增强

问题在于：
- 这些词条主要改变数值，不改变打法结构。
- 玩家拿到强化后的主要感知仍是“更快、更大、更高”，不是“这局已经变成另一种构筑”。

### 4.2 全局通用强词条会过早统一玩家选择
当前唯一全局被动是“全部神通冷却缩短”。
这类词条在早期极容易成为通吃最优解，原因很直接：
- 对任意 3 神通组合都有效
- 不依赖特定学校
- 没有明显副作用或取舍

如果泛用词条长期强于专属词条，玩家会更快收敛到同一选择路径。

### 4.3 各流派缺少“流派内被动”，只剩“流派名义区分”
当前没有看到这类层：
- 御剑系专属被动
- 火法系专属被动
- 冰法系专属被动
- 雷法系专属被动
- 心法/护体系专属被动

结果会是：
- 不同流派都在抢同一批全局泛用词条。
- 构筑差异主要来自起手神通，不来自中局成长路线。
- 2 次强化后，很多局会看起来只是“同一套 build 的不同数值档位”。

### 4.4 临时神通层缺失，会让构筑失去“转向时刻”
轻肉鸽的关键体验不是只把既有技能抬高，而是中局出现新的拼装关系。
当前临时神通层完全缺失，直接带来两个问题：
- 玩家无法在局内形成“副核神通”。
- 三选一无法承担“赌转型 / 补短板 / 做联动”的职责。

## 5. 局外成长与局内成长边界的风险点

### 5.1 文档边界清楚，但真源边界没有落到表结构
文档层面已经说清：
- 局外负责永久解锁与整备
- 局内负责本局强化

但表结构没有明确字段区分：
- 哪些是局外可装配基础神通
- 哪些是局内临时解锁神通
- 哪些是辅助道法
- 哪些只在本局结算前有效

当前 `轻肉鸽神通战斗配置总表.csv` 缺少“来源层级/获得方式/是否局内临时”字段，导致边界停留在文档，没沉到真源。

### 5.2 `神通配置总表.csv` 与轻肉鸽战斗表存在真源冲突
当前总神通表明确是：
- 每个大境界 1 个神通
- 共 14 个大境界神通

但 `轻肉鸽神通战斗配置总表.csv` 实际定义的是另一套战斗神通：
- 名称体系不同
- 学派体系不同
- 炼气阶段甚至出现 2 个技能（`御剑术`、`清心诀`）
- 与“每个大境界仅 1 个神通”的总规则并不直接对齐

这会造成一个关键问题：
- 玩家局外整备时，到底以哪张表为准？
- 轻肉鸽战斗表中的技能，到底是局外正式神通，还是局内临时神通，还是战斗化映射？

如果不先定真源映射，后面所有强化、解锁、装配规则都会漂移。

### 5.3 局内成长的写回与清空规则缺失
虽然文档已经说明“局内成长不写回永久数值”，但还缺少以下明确规则：
- 局内强化选择记录写到哪里
- 临时神通写到哪里
- 复活后是否保留本局已选强化
- 撤离、失败、胜利三种结算后，本局强化何时清空
- 局内成长是否需要进入结算页展示但不写回

这部分若不补，工程实现时一定会出现“数据该不该保留”的歧义。

## 6. 强化主表字段不足点

### 6.1 当前字段只够描述简单数值 buff
当前 `轻肉鸽局内强化配置总表.csv` 字段为：
- `upgradeId`
- `upgradeType`
- `rarity`
- `displayName`
- `targetType`
- `targetId`
- `description`
- `maxStack`
- `canRepeat`
- `unlockCondition`
- `valueA/valueB/valueC`

它能表达“某技能数值 +X”，但不足以表达更完整的轻肉鸽成长逻辑。

### 6.2 建议补足的字段类别
| 字段类别 | 建议字段 | 为什么需要 |
| --- | --- | --- |
| 池子控制 | `weight`、`offerGroup`、`schoolTag`、`buildTag` | 需要控制出现概率，避免三选一长期刷同质项 |
| 前置与互斥 | `requireTags`、`excludeTags`、`requireUpgradeIds`、`exclusiveGroup` | 需要支撑流派链路、转职链路、冲突词条 |
| 效果表达 | `effectKey`、`effectMode`、`valueTypeA/B/C` | 不能长期依赖 `valueA/B/C` 猜含义 |
| 目标层级 | `targetLayer`、`targetSchool` | 区分基础神通、临时神通、全局被动、辅助道法 |
| 临时神通解锁 | `grantSkillId`、`grantSkillSlotType`、`grantLimitType` | 否则“解锁额外临时神通”无法接表 |
| 时效与清空 | `durationType`、`clearTiming`、`persistScope` | 明确只在本局有效，还是持续到本局结算页 |
| 交互展示 | `titleText`、`descShort`、`descDetail`、`iconKey` | 当前只有一个描述字段，不够支撑三选一展示层 |
| 调试与排序 | `sortOrder`、`isEnabled`、`devNote` | 便于迭代期灰度、排序、屏蔽和排查 |

### 6.3 战斗神通主表也缺关键字段
当前 `轻肉鸽神通战斗配置总表.csv` 缺少至少以下判断字段：
- 是否局外可装配
- 是否局内临时获得
- 是否属于辅助道法
- 触发层级/优先级
- 与总神通表的映射字段
- 最大同时存在数 / 占用的战斗位类型

没有这些字段，就无法稳定区分：
- 基础神通
- 临时神通
- 辅助道法
- 被动触发技

## 7. 缺少的关键层

### 7.1 被动词条层明显偏薄
当前只有 1 条全局被动，远低于“构筑分化”所需的最小厚度。
建议至少补齐以下被动类别：
- 学派专属被动
- 触发类被动
- 生存类被动
- 资源循环类被动
- 条件换收益类被动

### 7.2 辅助道法层还没有独立出来
当前只有 `清心诀` 及其一条强化，辅助层仍像单点功能，不像一套可选系统。
建议至少把辅助道法定义为独立层，用于承担：
- 回复
- 护盾/护体
- 聚怪/控场辅助
- 增伤标记
- 灵气/施法节奏支持

### 7.3 临时神通层当前为 0
这是当前缺口里最关键的一项。
如果没有临时神通层：
- 三选一就没有“拿新能力”的惊喜点。
- 中途转型无法成立。
- 3 神通整备会过度决定整局上限，局内成长退化成只做微调。

## 8. 缺失项清单与建议

### 8.1 缺失项：展示文案
当前缺少至少以下文案真源：
- 三选一弹层标题
- 稀有度文案
- 已拥有/已满层/前置不足提示文案
- 临时神通解锁提示文案
- 放弃/不可选是否存在的文案规则

建议：
- 将“展示名、短描述、长描述、条件提示”从单一 `description` 中拆开管理。

### 8.2 缺失项：状态机细节
当前只有战斗大状态，没有局内强化子状态机。
至少缺少：
- 候选生成
- 展示锁定
- 玩家选择确认
- 结果应用
- 恢复战斗
- 重入保护

建议：
- 为 `upgrade` 单独补一个子状态流，避免后续把候选生成、应用、恢复混在一起。

### 8.3 缺失项：交互逻辑
当前还没有明确：
- 三个选项是否允许同目标不同词条并存
- 是否允许同名强化在同一次三选一里重复出现
- 词条满层后如何从池中剔除
- 临时神通满上限时如何处理
- 玩家选中后是否需要二次确认或详情预览

建议：
- 把“候选生成规则”和“选择后应用规则”拆成两个独立小节，不要混写。

### 8.4 缺失项：数据真源登记
总 PRD 当前正式数值真源列表中，还没有把以下两张表纳入正式真源：
- `轻肉鸽神通战斗配置总表.csv`
- `轻肉鸽局内强化配置总表.csv`

建议：
- 在总 PRD 的“数值与配置真源”中正式登记这两张表，否则实现阶段会出现“方案提了，但主控真源未备案”的口径冲突。

### 8.5 缺失项：写回规则
当前没有明确：
- 本局强化是否进入存档临时态
- 本局失败/撤离/胜利后清空时机
- 广告复活前后是否保留本局强化
- 临时神通是否参与结算展示

建议：
- 单独增加“局内成长存储与清空规则”小节，明确它只属于 run 级状态，不进入永久成长。

### 8.6 缺失项：依赖关系细化
当前文档只写了大依赖，没有写清：
- 总神通表与轻肉鸽战斗神通表的映射关系
- 局内强化表与 UI 三选一展示字段的对接关系
- 临时神通与战斗上限控制的依赖关系

建议：
- 先补“谁是局外真源、谁是局内战斗真源、两者如何映射”的责任表，再进入实现。

### 8.7 缺失项：微信运行态验收细化
当前只有原则性验收，还缺少局内强化专属的运行态验收项，例如：
- `upgrade` 弹出时战斗是否真正暂停
- 选择后是否稳定恢复，不重复弹出
- 低性能场景下弹层与战斗恢复是否有明显卡顿
- 2~3 次强化后是否真能感知流派差异
- 临时神通加入后是否触发性能/表现异常

建议：
- 在微信开发者工具验收总表里新增“局内强化”专项 case，而不是只保留一句体验性描述。

## 9. 对主控的 3 条最关键建议

### 建议 1
先统一“总神通表”和“轻肉鸽战斗神通表”的真源关系，再谈强化池扩容。
否则局外可装配神通、局内临时神通、辅助道法三层会一直混在一起，后面所有字段补充都会失焦。

### 建议 2
把局内强化池从“6 条演示词条”提升到“可支撑 2 次强化后明显分化”的最小厚度，优先补：
- 临时神通解锁层
- 学派专属被动层
- 每个已上表神通至少 2~3 条专属成长词条

### 建议 3
在进入实现前，先补完 4 个缺口：
- 正式真源登记
- 局内成长写回/清空规则
- `upgrade` 子状态机
- 微信运行态专项验收

这 4 项不补，后续最容易出现“代码先跑了，但文档真源和运行态口径全不一致”的返工。
