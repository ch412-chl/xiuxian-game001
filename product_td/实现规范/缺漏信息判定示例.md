# 缺漏信息判定示例

## 1. 文档定位
本文档用于给实现线程一个非常具体的判断标准：
**哪些情况算“信息足够，可以写代码”，哪些情况算“缺信息，只能报缺口”。**

## 2. 一句话原则
只要缺少以下任一项：
- 标题
- 文案
- 状态机
- 交互逻辑
- 数据真源
- 写回规则
- 依赖关系
- 微信运行态验收

就不能直接进入编码。

## 3. 可直接编码示例

### 示例 A：轻肉鸽战场进入首领波
已知：
- 标题：有
- 状态机：`wave(19) -> boss(20)`
- 文案：有首领名称与机制摘要
- 数据真源：有首领机制表
- 写回规则：无持久写回
- 依赖关系：依赖第 20 波首领生成触发
- 运行态验收：`WT-LR-09`

结论：
- 信息完整
- 可以编码

### 示例 B：广告撤离
已知：
- 触发页面：轻肉鸽战场页
- 触发条件：无传送符且当前状态允许主动撤离
- 成功结果：进入 `settle_preview -> settle_commit`
- 写回规则：保留 runBag 并在 `settle_commit` 写入 inventory
- 运行态验收：`WT-LR-09`

结论：
- 信息完整
- 可以编码

## 4. 不可直接编码示例

### 示例 C：想做“战场事件节点”
目前只有一句话：
- “局内随机补给 / 小强化 / 临时风险，统一视为战场内事件节点”

缺失：
- 标题/页面定位
- 状态机
- 按钮显隐
- 数据真源
- 写回规则
- 微信运行态验收

结论：
- 只能报缺口
- 只能给出补充建议
- 不得直接新增页面或代码逻辑

### 示例 D：想给精英新增独立图鉴
目前已有：
- 精英配置表
- 图鉴表

但缺失：
- 是否作为一级图鉴条目
- 解锁时点
- 图鉴页展示层级
- 洞府摘要如何统计
- 微信运行态验收

结论：
- 只能报缺口
- 不能因为“表里已有 eliteId”就直接实现

## 5. 报缺口模板
建议实现线程在遇到缺信息时统一用以下格式：

- 功能：
- 计划改动：
- 缺失项：
- 为什么当前不能直接编码：
- 建议补充：
- 影响范围：

## 6. 最后提醒
**“看起来大概知道怎么做”不算信息完整。**
只有当实现线程能明确回答：
- 谁触发
- 何时触发
- 显示什么
- 写到哪里
- 依赖谁
- 怎么验

才允许开工。
